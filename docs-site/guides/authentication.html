The Pulse Pager REST API authenticates with per-org, role-scoped API keys. A key belongs to one org and can do exactly what its role can do, never more.

Creating a key

In the app, open your org settings and create an API key. You give it a name and a role. A key's role is member or admin only; keys are never owner-equivalent, so a leaked key cannot change billing, transfer ownership, or delete the org.

The secret is shown once. The full secret (pulse_sk_...) is returned exactly once, at creation, and never again. Pulse Pager stores only a hash and a short non-secret prefix. Copy the secret straight into your secret manager or CI secret store.

You can also create a key through the API itself (admin role required):

curl -X POST https://api.pulsepager.com/api/v1/orgs/<orgId>/api-keys \
  -H "Authorization: Bearer pulse_sk_an_existing_admin_key" \
  -H "Content-Type: application/json" \
  -d '{"name":"ci-deploy","role":"member"}'

The response carries the key metadata plus the one-time secret. A later GET .../api-keys returns metadata only (name, role, prefix, created-by, created, last-used), never the secret.

Using a key

Send the full secret as a bearer token in the Authorization header on every request: 

Authorization: Bearer pulse_sk_<secret>

The server hashes the presented value and compares it against stored hashes. For example:

curl https://api.pulsepager.com/api/v1/orgs/<orgId>/monitors \
  -H "Authorization: Bearer pulse_sk_your_secret_here"

What the responses mean

StatusMeaning
200 / 201 / 204The call succeeded.
401Missing, malformed, unknown, or revoked credential. The server does not know who you are.
403Known key, but its role is too low for this operation, or the org's plan does not allow it (for example, writes on a read-only Free plan).
422Validation failed. The error body carries a fields map naming each bad field.

The distinction between 401 (who are you) and 403 (you are known but not allowed) is kept strict. Every error uses the standard envelope: 

{ "error": { "code": "forbidden", "message": "Human-readable summary." } }

Roles and what a key can do

A key inherits a role and is checked against the same permission matrix a human of that role would be, restricted to the API surface.

  • member: view and manage monitors, channels, and status pages; run check-now; view and annotate incidents.
  • admin: everything a member can do, plus manual incident close, managing people and keys, and viewing billing.

Owner-only actions (managing billing, transferring ownership, deleting the org) have no API and stay in the UI by design.

Revoking a key

Any key can be revoked immediately. A revoked key fails the very next request with 401. The key list shows a last-used timestamp so stale keys are easy to spot and rotate. 

curl -X DELETE https://api.pulsepager.com/api/v1/orgs/<orgId>/api-keys/<id> \
  -H "Authorization: Bearer pulse_sk_an_admin_key"

Webhooks

Org-level outbound webhooks are a programmatic event feed (monitor.down, monitor.recovery, incident.opened, incident.closed) so you can ingest Pulse Pager events into your own pipeline without polling. Register and manage them under /orgs/<orgId>/webhooks (owner or admin); the signing secret is returned once when you create the webhook and can be rotated.

Each delivery is signed with the webhook's secret. The X-Pulse-Signature header is t=<unix>,v1=<hex hmac-sha256>, where the HMAC is computed over <t>.<raw body>: recompute it with your secret and compare. The timestamp is part of the signed input, so a replayed delivery with a stale t will not verify. A stable event_id (also sent as the X-Pulse-Event-Id header) lets retries dedup safely.

Read the full set of operations and schemas in the API reference.
