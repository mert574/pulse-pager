# ADR-0012: API key authentication via SHA-256 and a Redis cache

Status: Accepted
Date: 2026-06-21
Deciders: Architecture
Related: RFC-000 section 7.3, RFC-003 sections 5.1 and 5.2, RFC-012 section 5.3, CONSISTENCY-REVIEW F-1 and F-4, ADR-0005

## Context
Scripts authenticate to the public API with a per-org API key carried as a bearer token. Keys are shown once, stored at rest as a hash, and verified on a high-rate path. The secret is generated by us as a uniform-random value, not chosen by a human, so it has no offline brute-force surface the way a password does.

## Options considered
- SHA-256 of the random secret, verified through a Redis cache - fast on the high-rate verify path, and a uniform-random 128-bit secret has nothing to brute-force offline so a fast hash is safe here.
- bcrypt or argon2 - the right choice for human passwords, but a deliberately slow hash burns CPU on every verify for zero security gain when the input is already high-entropy random.

## Decision
API keys use the canonical prefix pulse_sk_ followed by a 128-bit (22+ char) uniform-random secret. At rest the key is stored as a SHA-256 of the presented secret, plus a non-secret prefix and a last-used timestamp. Verification goes through a Redis cache. Keys are role-scoped to member or admin only (never owner) and revoke immediately. bcrypt is retained only for the self-host bootstrap admin password, not for API keys.

## Consequences
Verification stays cheap on a path that can run hot, and storing a SHA-256 rather than the plaintext means a database leak does not expose usable keys (there is nothing to reverse in a 128-bit random preimage). Because owner is never an allowed key role, a leaked key cannot automate billing or ownership. The cost is that the cache is on the verify path, so a Redis outage degrades verify latency until it warms again. This is the canonical prefix across the doc set (the consistency review fixed pulse_live_ stragglers to pulse_sk_ and the argon2id note in RFC-001 to SHA-256). Revisit only if API keys ever become user-chosen secrets, which would reintroduce a brute-force surface.
