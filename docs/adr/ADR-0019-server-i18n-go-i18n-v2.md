# ADR-0019: Server-side i18n library for Go is go-i18n v2

Status: Accepted
Date: 2026-06-22
Deciders: Architecture
Related: RFC-014 sections 10.2, 10.4, 6.2, ADR-0018

## Context
Server-generated content that must localize (notification bodies, server-rendered strings for non-UI consumers, the `message` field rendered in the resolved locale) needs a Go i18n library. The workflow goal is catalog-additions-only: adding a locale should be dropping in a per-locale message file, not a codegen and rebuild step. The build is `CGO_ENABLED=0` static/distroless (RFC-000 section 3), so a cgo native dependency is out.

## Options considered
- `github.com/nicksnyder/go-i18n/v2` (chosen). Built around CLDR pluralization and a message catalog loaded from files at startup, with named template data and an explicit fallback chain ending at `en`. Supports ICU-style plural rules, no cgo, mature and widely used. Fits "render `message` in the resolved locale, fall back to `en`" directly and matches the catalog-additions-only workflow.
- `golang.org/x/text` message/catalog - rejected. Lower-level and oriented to compile-time catalogs generated by `gotext`, which couples adding a locale to a codegen + rebuild step. Good pluralization, but the workflow is heavier than a file-loaded catalog and fights the additions-only goal.
- A pure ICU binding (cgo ICU) - rejected. Full ICU correctness, but cgo breaks the static/distroless build and adds a heavy native dependency for a need go-i18n already meets. Revisit only on an ICU feature go-i18n cannot express.

## Decision
The server uses `github.com/nicksnyder/go-i18n/v2` for localization. Server catalogs are ICU-message files loaded per locale (one file per locale, for example `active.en.toml` / `active.de.toml`), keyed by the same dotted codes the localizable-string contract uses (ADR-0018). The `en` catalog is the source of truth on the server side; resolution falls back to `en`. Adding a locale is adding `active.<locale>.*` files, no code change.

## Consequences
Adding a server locale stays a file drop with no rebuild of generated catalogs and no cgo, keeping the distroless build intact. The server and FE key off the same code namespace, so a code means the same string on both sides; a CI check asserts every emitted code resolves in both catalogs or falls back to `en`. The cost is carrying go-i18n's catalog-loading at startup and keeping the TOML message files in step with the FE XLIFF catalogs (the shared code namespace is the contract that keeps them aligned). Revisit only if an ICU feature go-i18n cannot express becomes load-bearing.
